# PRD v0.1

## Product: High-Performance Workflow Builder SDK

### 문서 상태

Draft

### 문서 목적

노드 기반 SaaS, 자동화 툴, 내부 운영툴을 만드는 팀이 **복잡한 워크플로우 UI를 빠르게 구축하고**, 규모가 커져도 **성능 저하 없이 운영 가능한 빌더**를 제공하기 위한 제품 요구사항 정의.

---

## 1. 제품 한 줄 정의

**복잡한 워크플로우도 버벅이지 않게 만들 수 있는 production-ready workflow builder SDK**

보조 설명:

* React 앱에 쉽게 붙일 수 있는 워크플로우 빌더 SDK
* 대규모 노드/엣지 환경에서도 성능 저하를 최소화
* 팀이 캔버스 엔진을 처음부터 만들지 않고도 서비스 수준의 에디터를 출시 가능

---

## 2. 배경 및 문제 정의

많은 팀이 노드 기반 제품을 만들 때 다음 흐름을 겪는다.

1. 처음에는 React Flow 같은 범용 라이브러리로 빠르게 프로토타입한다.
2. 노드 수가 늘고, 커스텀 인터랙션이 많아지고, 설정 패널/검증/히스토리/대형 캔버스 요구가 붙는다.
3. 성능 병목과 구조적 한계가 생긴다.
4. 결국 직접 캔버스, 렌더링, 가상화, 인터랙션을 다시 만들거나 억지로 유지보수한다.

이때 팀이 진짜 원하는 것은 “렌더링 기술”이 아니라 아래다.

* 서비스에 붙일 수 있는 수준의 워크플로우 빌더
* 커스텀이 쉽고
* 대규모 그래프에서도 느려지지 않으며
* 출시까지 시간을 줄여주는 기반

즉 시장의 문제는 “빠른 엔진이 없다”가 아니라
**“프로덕션에 견디는 워크플로우 빌더를 빠르게 만들 수 있는 기반이 부족하다”**이다.

---

## 3. 제품 가설

### 핵심 가설

노드 기반 제품을 만드는 팀은 “기술적으로 멋진 그래프 엔진”보다
**“실제로 붙여서 바로 제품을 만들 수 있는 빌더 SDK”**를 더 높은 가치로 인식한다.

### 서브 가설

* 성능은 구매 이유라기보다 **도입 정당화 요소**다.
* 메인 메시지는 “무엇을 만들 수 있는가”여야 한다.
* Rust/WASM/Canvas는 메인 카피가 아니라 **신뢰를 보강하는 기술 근거**로 작동한다.
* 초기 고객은 최종 사용자가 아니라 **제품팀/프론트엔드팀**이다.

### 메시지 우선순위

1. 무엇인지: 워크플로우 빌더 SDK
2. 누구를 위한지: 노드 기반 제품을 만드는 팀
3. 왜 써야 하는지: 복잡해져도 느려지지 않고, 출시가 빨라짐

## 7. 제품 목표

### 비즈니스 목표

* 노드 기반 제품 팀이 검토 가능한 **명확한 카테고리**를 만든다.
* “React Flow 대체재”로 발견되되, 단순 대체가 아니라 **상위 개념의 제품**으로 인식된다.
* 초기 유저가 “빠르다”보다 “바로 붙일 수 있다”를 이유로 도입하게 만든다.

### 사용자 목표

* 첫 에디터를 30분 내 띄울 수 있다.
* 1주 내 제품에 맞는 기본 커스터마이징이 가능하다.
* 그래프가 커져도 실사용 가능한 상호작용 성능을 유지한다.

### 제품 목표

* SDK + Starter Kit 조합으로 **도입 속도**와 **성능 신뢰성**을 동시에 제공한다.
* React 친화적 API로 진입 장벽을 낮춘다.
* 성능 최적화는 기본값으로 내장한다.

* 워크플로우 실행 엔진 자체

* 비개발자용 완전한 노코드 플랫폼

즉 이 제품은 **“워크플로우 UI를 만드는 기반”**이지, 워크플로우 제품 전체를 대체하는 것은 아니다.

## 9. 제품 원칙

### 1. Product-first, engine-second

겉으로는 SDK/Starter Kit, 안쪽에서 엔진이 작동한다.

### 2. Production-ready by default

데모용 캔버스가 아니라 실제 서비스에 넣을 수 있는 구조를 우선한다.

### 3. Integrate, don’t dictate

호스트 앱의 인증, 데이터 모델, 패널, 디자인 시스템과 자연스럽게 통합되어야 한다.

### 4. Performance at scale

소규모에서만 빠른 것이 아니라, 복잡도가 올라갈수록 가치가 커져야 한다.

### 5. Fast path to shipping

개발팀이 캔버스 기반부터 새로 짜지 않게 해야 한다.

## 10. 제품 구성

이 제품은 두 레이어로 구성한다.

### A. Core SDK

개발자가 React 앱 안에서 워크플로우 에디터를 만들 수 있게 해주는 핵심 라이브러리

포함 범위:

* 캔버스 렌더링
* 줌/패닝/선택/드래그
* 노드/엣지 데이터 모델
* 이벤트 시스템
* 커스텀 노드 렌더링 훅
* 상태 변경 훅
* 성능 최적화 레이어

### B. Starter Kit

서비스에 빠르게 붙이기 위한 UI 뼈대와 예제

포함 범위:

* 기본 레이아웃
* 노드 팔레트
* inspector panel
* toolbar
* mini map / selection tools
* 샘플 노드 세트
* persistence example
* docs + example apps

즉 **SDK가 코어 제품**, **Starter Kit이 도입 가속 장치**다.

## 11. 핵심 사용자 시나리오

### 시나리오 1

자동화 SaaS 팀이 React 앱에 워크플로우 편집기를 추가한다.

성공 조건:

* 설치 후 빠르게 기본 에디터를 띄운다.
* 회사 디자인 시스템에 맞는 노드 UI를 입힌다.
* JSON 저장/불러오기를 연결한다.

### 시나리오 2

기존 React Flow 기반 제품이 그래프 규모 증가로 성능 문제를 겪는다.

성공 조건:

* 기존 데이터 모델을 큰 수정 없이 이관 가능하다.
* 대규모 그래프에서 상호작용 품질이 개선된다.
* 커스텀 노드 로직을 유지할 수 있다.

### 시나리오 3

내부 운영툴 팀이 승인 프로세스/업무 플로우 빌더를 만든다.

성공 조건:

* 빠른 구축
* 읽기/편집 모드 지원
* 비즈니스 룰 검증과 상태 표시가 쉽다.

## 14. 상세 기능 요구사항

## 14.1 개발자 경험

### 요구사항

* 개발자는 30분 안에 기본 워크플로우 에디터를 띄울 수 있어야 한다.
* 커스텀 노드는 React 컴포넌트 형태로 정의 가능해야 한다.
* 호스트 앱의 상태관리 라이브러리와 함께 쓸 수 있어야 한다.
* 외부 데이터 패널, 모달, 설정 폼과 쉽게 연동 가능해야 한다.

### 수용 기준

* quickstart 문서만으로 샘플 에디터 구성 가능
* 최소 하나의 example repo 제공
* custom node, custom edge, custom panel 예제가 있어야 함

---

## 14.2 에디터 경험

### 요구사항

* 기본 인터랙션은 별도 설정 없이 자연스럽게 동작해야 한다.
* 노드 선택, 이동, 연결 작업이 끊김 없이 수행되어야 한다.
* 큰 그래프에서도 화면 조작이 실사용 수준으로 유지되어야 한다.

### 수용 기준

* 1,000개 노드 / 2,000개 엣지에서 패닝/줌/선택이 부드럽게 동작
* 5,000개 노드 규모에서도 탐색과 선택이 usable 해야 함
* 빈 캔버스, 대형 캔버스 모두 UX가 무너지지 않아야 함

## 15. 비기능 요구사항

## 15.1 성능

* 대규모 그래프 탐색 시 프레임 드랍이 체감상 적어야 한다.
* 노드 수 증가에 따라 DOM 기반 병목이 최소화되어야 한다.
* 대량 렌더링에서도 인터랙션 우선순위가 유지되어야 한다.

### 목표 지표

* 중간급 개발 환경에서 1k 노드 기준 주요 인터랙션이 55fps 내외
* 5k 노드에서도 주요 탐색 작업이 사용 가능 수준 유지
* 대규모 그래프에서 뷰포트 이동 시 심각한 입력 지연 없음

## 15.2 안정성

* 에디터가 host app 전체 성능을 저해하지 않아야 한다.
* 예외 상황에서 캔버스 전체가 깨지지 않고 복구 가능해야 한다.

## 15.3 접근성

* 기본 단축키 제공
* 키보드 기반 선택/삭제 지원
* 확대/축소 컨트롤 명시
* 캔버스 기반 구조의 한계를 고려한 최소 접근성 가이드 제공

## 15.4 호환성

* 최신 React / TypeScript 지원
* 주요 Chromium 계열 브라우저 우선 지원
* Safari/Firefox는 점진 지원 범위 명시

## 16. UX 원칙

* 기본 레이아웃은 익숙해야 한다.
* 상호작용은 “툴 같아야” 한다.
* 복잡한 그래프에서도 사용자가 길을 잃지 않아야 한다.
* 성능 최적화는 보여주기용이 아니라 “신뢰감”으로 느껴져야 한다.
* UI는 제품에 녹아들어야지, 외부 위젯처럼 보이면 안 된다.

### 경쟁 우위 메시지

1. 워크플로우 빌더 SDK
2. 대규모 환경에서 강함
3. React Flow 대안으로 고려 가능

## 19. 기술 방향성

이 섹션은 외부 마케팅보다 내부 정렬용이다.

### 내부 기술 원칙

* 성능 병목이 많은 부분은 Rust + WASM + Canvas 기반으로 처리
* React는 개발자 경험과 제품 통합 레이어 역할
* 캔버스 렌더링과 제품 UI 레이어를 분리
* 향후 headless API와 higher-level Starter Kit을 동시에 운영 가능한 구조 지향

### 외부 노출 원칙

* 기술 스택은 proof point로 쓰되, headline으로 쓰지 않는다.
* “왜 빠른가”는 서브 설명에 둔다.
* “무엇을 만들 수 있는가”를 항상 먼저 말한다.

## 22. 오픈 이슈

* OSS와 상용 기능의 경계는 어디까지 둘 것인가
* Starter Kit을 무료로 둘지, Pro로 둘지
* headless API 우선인지, batteries-included UI 우선인지
* 마이그레이션 대상 고객을 적극 공략할지, 신규 빌더 팀을 먼저 노릴지
* 그룹/서브플로우 기능을 MVP에 넣을지 P1로 미룰지
* 협업 기능의 우선순위를 언제 올릴지

## 23. 최종 결정 문장

이 제품의 정체성은 아래 문장으로 내부 합의를 맞춘다.

**우리는 렌더링 기술을 파는 것이 아니라, 성능 문제 없이 워크플로우 제품을 만들고 출시하게 해주는 기반을 판다.**

따라서 제품 정의는 다음과 같다.

**Main:** 고성능 워크플로우 빌더 SDK  
**Secondary:** 빠른 도입을 위한 Starter Kit  
**Support:** Rust + WASM + Canvas 기반 고성능 렌더링 아키텍처
