# PRD 업데이트 초안

## Workflow Builder SDK / Workflow Optimization UI

### 1. 문서 목적

현재 구현 중인 Workflow Optimization UI의 PRD를 기술 중심 설명에서 사용자 이해 중심 설명으로 재정렬한다.

지금 상태의 문제는 기능이 부족해서가 아니다. WASM + Rust 기반 엔진과 UI는 기능적으로 충분히 강하다. 하지만 제품을 보여줄 때 사용자가 “그래서 이게 정확히 어떤 제품인지”를 빠르게 이해하지 못한다.
이번 PRD 업데이트의 목적은 제품의 1차 인지 포인트를 엔진, 서버, 호스트 연동이 아니라 **노드 선택 → inspector/form 편집 → 노드 UI 즉시 반영**이라는 사용자 경험으로 바꾸는 것이다.

---

### 2. 배경 및 문제 정의

현재 제품은 내부적으로 다음 요소를 이미 갖추고 있다.

* WASM + Rust 기반 워크플로우 처리 엔진
* 커스텀 노드 렌더링
* 클릭/선택 등 기본 상호작용
* 호스트 앱 내 통합 가능성

그런데 데모와 제품 설명이 엔진 구조, 서버 연결, same-server host integration view 같은 내부 관점에서 출발하면서 사용자는 아래를 직관적으로 이해하지 못한다.

* 이 제품이 React Flow 대체재인지
* 사용자가 어떤 방식으로 워크플로우를 편집하는지
* 캔버스와 폼이 어떻게 연결되는지
* 실제 제품 안에서 어떤 경험으로 쓰이게 되는지

즉, 현재의 핵심 문제는 기능 구현이 아니라 **제품 가치가 사용자 경험으로 번역되지 않는 문제**다.

---

### 3. 제품 방향 재정의

기존에는 제품이 “좋은 캔버스 데모” 혹은 “엔진이 잘 도는 워크플로우 UI”처럼 보였다.
앞으로는 다음처럼 정의되어야 한다.

**우리가 만들고 보여줘야 하는 것은 React Flow를 대체할 수 있는 Workflow Builder SDK다.**

사용자가 처음 이해해야 하는 한 문장은 아래와 같다.

> 이 제품은 노드 기반 워크플로우를 앱 안에 임베드하고, 우측 폼으로 값을 편집하면 그래프 상태와 노드 UI가 즉시 연결되는 Workflow Builder SDK다.

---

### 4. 핵심 문제 진술

현재 데모는 내부 구현이 너무 앞에 드러나서, 사용자가 제품의 핵심 가치인 **노드 + 폼 + 상태 연결**을 바로 이해하지 못한다.

따라서 이번 단계의 해결 원칙은 다음과 같다.

* 데모의 시작점을 엔진 설명이 아니라 편집 경험으로 옮긴다.
* 서버/포트/WASM/Rust는 기본적으로 사용자 인지 대상에서 숨긴다.
* 첫 번째 proof는 host integration view가 아니라 **form-driven custom node editing**이어야 한다.
* dev server와 SDK 사용 경험을 명확히 분리한다.

---

### 5. 목표

#### 제품 목표

사용자가 첫 시연에서 이 제품을 “React Flow 대체 가능한 Workflow Builder SDK”로 이해하도록 만든다.
사용자가 노드 편집 경험을 직관적으로 따라갈 수 있게 만든다.
기술 구조를 몰라도 제품 가치를 설명할 수 있게 만든다.

#### UX 목표

노드를 클릭하면 바로 편집 가능한 컨텍스트가 열린다.
사용자는 폼 변경이 그래프와 노드 UI에 즉시 반영된다는 인과관계를 자연스럽게 느낀다.
제품의 학습 비용을 엔진 이해가 아니라 UI 사용 이해 수준으로 제한한다.

#### 데모 목표

첫 1~2분 안에 사용자가 “무엇을 하는 제품인지” 설명할 수 있어야 한다.
첫 proof에서 host integration보다 form-driven editing이 중심에 와야 한다.

---

### 6. 비목표

이번 단계에서 아래 항목은 메인 데모 핵심이 아니다.

* same-server host integration view를 메인 proof로 삼는 것
* 서버/포트 구조를 사용자에게 설명하는 것
* WASM/Rust 엔진 구조를 전면에 내세우는 것
* persistence를 필수 플로우로 전면 노출하는 것

이 요소들은 필요할 수 있지만, 이번 단계에서는 보조적 구현 디테일 또는 후속 확장 항목으로 취급한다.

---

### 7. 대상 사용자

#### 1차 사용자

제품 안에 워크플로우 빌더를 임베드하려는 프론트엔드 팀
기존 React Flow 기반 편집기를 더 제품 친화적인 방식으로 바꾸려는 팀
노드 렌더링, inspector, 상태 연결을 커스터마이즈하려는 팀

#### 2차 사용자

내부 PM, 디자이너, 솔루션 팀
기술 시연을 보고 도입 가능성을 판단하는 이해관계자

---

### 8. 핵심 사용자 가치

사용자가 실제로 느껴야 하는 가치는 다음과 같다.

* 노드를 제품 UI에 맞게 렌더링할 수 있다.
* 노드를 선택하면 자연스럽게 inspector/form이 열린다.
* 폼 값을 수정하면 노드 UI와 그래프 상태가 즉시 바뀐다.
* 필요하면 persistence를 연결할 수 있다.
* 내부 엔진, 서버 포트, WASM 런타임을 몰라도 사용할 수 있다.

즉, 제품의 핵심 가치는 그래프 캔버스 자체가 아니라 **캔버스와 폼이 연결된 편집 경험**이다.

---

### 9. 제품 원칙

첫째, 제품은 실행 엔진보다 편집 경험으로 먼저 이해되어야 한다.
둘째, 사용자 편집의 중심은 inspector/form이다.
셋째, 폼 변경은 노드 UI와 그래프 상태에 즉시 반영되어야 한다.
넷째, 엔진 구조는 기본 UX에 드러나지 않아야 한다.
다섯째, SDK는 개발자가 자기 제품 안에 쉽게 붙일 수 있는 형태여야 한다.

---

### 10. 핵심 사용자 시나리오

대표 시나리오는 **Form-driven Custom Node Editing**이다.

1. 사용자가 워크플로우 빌더 화면에 진입한다.
2. 사용자가 특정 노드를 클릭한다.
3. 우측 inspector/form 패널이 열린다.
4. 사용자가 텍스트, 옵션, 숫자 등 노드 속성을 수정한다.
5. 수정값이 노드 UI에 즉시 반영된다.
6. 그래프 상태도 동기화된다.
7. 사용자는 이 제품이 폼과 그래프가 연결된 워크플로우 편집기라는 모델을 이해한다.

이 시나리오가 다음 proof이자 첫 번째 데모 스크립트가 되어야 한다.

---

### 11. 기능 요구사항

#### 11.1 노드 선택 및 편집 진입

사용자는 노드를 클릭하거나 선택할 수 있어야 한다.
선택된 노드에 대응하는 inspector/form이 우측 패널 또는 명확한 보조 편집 영역에 표시되어야 한다.

#### 11.2 Form-driven editing

편집 UI는 react-hook-form과 유사한 구조 또는 그에 준하는 선언적 폼 패턴을 지원해야 한다.
노드별로 다른 필드 구성이 가능해야 한다.
커스텀 노드 타입마다 전용 편집 스키마를 연결할 수 있어야 한다.

#### 11.3 상태 동기화

폼 값 변경 시 노드 UI가 즉시 반영되어야 한다.
폼 상태와 그래프 상태 사이의 연결이 일관되어야 한다.
외부 상태 관리와 연결 가능한 컴포넌트 API가 있어야 한다.

#### 11.4 커스텀 노드 렌더링

노드 UI는 제품 디자인 시스템에 맞춰 커스터마이즈 가능해야 한다.
노드 내부 표시값은 폼 변경 결과를 시각적으로 보여줘야 한다.

#### 11.5 임베드 가능한 컴포넌트 API

제품은 사용자가 아래와 같이 인식할 수 있어야 한다.

```tsx
<WorkflowBuilder
  nodes={nodes}
  edges={edges}
  onNodesChange={...}
  onEdgesChange={...}
/>
```

즉, 사용자는 이 제품을 “엔진을 띄우는 시스템”이 아니라 “붙여서 쓰는 SDK 컴포넌트”로 이해해야 한다.

#### 11.6 persistence 연결 가능성

저장/불러오기 또는 외부 persistence는 연결 가능해야 한다.
다만 첫 proof에서는 필수적으로 전면 노출하지 않는다.
persistence는 선택적 확장 계층으로 다룬다.

---

### 12. UX 요구사항

첫 화면 또는 첫 데모에서 기술 용어보다 편집 가능한 결과물이 먼저 보여야 한다.
inspector/form은 노드 선택의 자연스러운 연장처럼 느껴져야 한다.
폼 변경 결과는 추가 설명 없이도 시각적으로 분명해야 한다.
빈 캔버스보다 예시 워크플로우가 기본 상태로 제공되는 것이 바람직하다.
사용자는 서버 연결 여부, wasm 포트, 런타임 구조를 알지 못해도 제품을 조작할 수 있어야 한다.
데모 스크립트는 “노드 선택 → 값 수정 → 즉시 반영” 순서로 고정한다.

---

### 13. 기술/아키텍처 원칙

유지할 것은 다음과 같다.

* WASM + Rust 기반 처리 엔진
* 내부 실행용 dev server
* host integration 가능 구조
* custom node architecture

기본 UX에서 숨길 것은 다음과 같다.

* 서버/포트 개념
* 내부 엔진 실행 방식
* same-server 여부
* 런타임 복잡성

핵심 분리 원칙은 명확하다.

* dev server는 내부 개발 및 예제 실행 도구다.
* SDK 사용 경험은 사용자가 실제로 접하는 제품이다.

PRD와 데모 구조는 이 둘을 반드시 분리해서 다뤄야 한다.

---

### 14. 이번 단계의 우선순위

기존 추천 우선순위는 `host integration view`였다.
이번 업데이트 이후의 우선순위는 `form-driven custom node editing proof`다.

이번 단계의 산출물은 “호스트와 잘 연결된다”는 얇은 기술 시연이 아니라,
**“노드와 폼이 자연스럽게 연결된 제품 경험”**을 보여주는 proof여야 한다.

---

### 15. 데모/프로토타입 범위

#### P0. Form-driven custom node editing proof

반드시 포함되어야 할 흐름은 다음과 같다.

1. 예시 워크플로우 로드
2. 노드 선택
3. 우측 inspector/form 오픈
4. 값 수정
5. 노드 UI 즉시 반영
6. 그래프 상태 변경 확인

#### P1. 디자인 시스템 적합성

버튼, 입력 필드, 패널 구조를 제품 디자인 시스템에 맞게 치환 가능함을 보여준다.

#### P2. persistence 예시

저장/불러오기 또는 외부 상태 저장 연결 예시를 추가한다.
다만 첫 제품 이해를 방해하지 않는 보조 단계로 둔다.

#### P3. host integration / same-server thin slice

필요 시 기술 검증용으로 유지하되, 메인 데모 서사의 앞단에는 두지 않는다.

---

### 16. 성공 지표

첫 시연 후 사용자가 제품을 한 문장으로 설명할 수 있어야 한다.
사용자가 이 제품을 워크플로우 엔진이 아니라 워크플로우 빌더/편집 SDK로 인식해야 한다.
첫 편집 완료까지 걸리는 시간이 짧아야 한다.
별도 설명 없이도 노드 선택과 폼 편집 흐름을 따라갈 수 있어야 한다.
내부/외부 평가자가 React Flow 대체 가능성에 대해 긍정적으로 प्रतिक्रिया해야 한다.
제품 가치 설명에서 엔진/서버가 아니라 편집 경험이 먼저 언급되어야 한다.

---

### 17. 수용 기준

다음 조건을 만족하면 이번 단계의 목표를 달성한 것으로 본다.

* 커스텀 노드를 선택하면 대응하는 폼이 열린다.
* 폼 변경이 노드 UI에 즉시 반영된다.
* 그래프 상태와 폼 상태의 연결이 명확하다.
* 사용자는 내부 서버/엔진 구조를 몰라도 데모를 이해할 수 있다.
* 제품 설명 시 “노드 + 폼 + 상태 연결”이 핵심 가치로 전달된다.
* host integration 설명 없이도 제품 가치가 성립한다.

---

### 18. 리스크

엔진 중심 시연 습관이 남아 있으면 다시 기술 데모처럼 보일 수 있다.
폼 편집 UX가 약하면 React Flow 대체제라는 메시지가 약해진다.
persistence나 host integration를 너무 일찍 노출하면 핵심 인지 흐름이 깨질 수 있다.
WASM/Rust 구현 성과를 과도하게 강조하면 제품 이해가 다시 어려워질 수 있다.

---

### 19. 한 줄 결론

이번 PRD 업데이트의 핵심은 제품을 **WASM/Rust 기반 워크플로우 엔진 데모**가 아니라
**React Flow를 대체할 수 있는 form-driven Workflow Builder SDK**로 재정의하는 것이다.

다음 proof의 기준은 host integration이 아니라,
**노드 선택 → inspector/form 편집 → 노드/그래프 상태 즉시 반영**이 자연스럽게 보이는지 여부다.
